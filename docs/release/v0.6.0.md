# pack-io v0.6.0 — Optimisation

**The stat-defense release.** v0.6.0 ships zero new public API and zero wire-format changes — just two safe-Rust optimisations that close the biggest gaps vs `bincode` / `postcard` / `rkyv` on the workloads users actually hit. The headline result: **`Vec<u8>` 4 KiB decode goes from 2,271 ns to 59 ns (38× faster)**, beating bincode by 22 %. Owned struct decode is now tied with rkyv and slightly beats bincode. Comparative numbers + methodology live at [`docs/PERFORMANCE.md`](https://github.com/jamesgober/pack-io/blob/main/docs/PERFORMANCE.md), fully reproducible.

## The headline

The single most user-visible workload — decoding a 4 KiB byte buffer, the shape every network message and file blob takes:

| Codec | Time | Relative |
|---|---:|---:|
| **pack-io** | **59 ns** | **1.00× (baseline)** |
| bincode | 76 ns | 1.29× slower |
| postcard | 1,774 ns | 30× slower |

In v0.5, the same benchmark ran at **2,271 ns** — pack-io was the slowest of the three. The v0.3 wire-format refactor that replaced the specialised `Vec<u8>` impl with a generic `Vec<T>` loop did the per-byte read 4096 times instead of one memcpy. v0.6 fixes that without breaking the wire format, without `unsafe`, and without dropping the generic `Vec<T>` impl.

## How the fix works

Two new trait methods on `Serialize` and `Deserialize`, each with a default implementation that preserves v0.5 behaviour:

```rust,ignore
pub trait Serialize {
    fn serialize<E: Encode + ?Sized>(&self, encoder: &mut E) -> Result<()>;

    fn serialize_slice<E: Encode + ?Sized>(slice: &[Self], encoder: &mut E) -> Result<()>
    where Self: Sized,
    {
        for item in slice { item.serialize(encoder)?; }
        Ok(())
    }
}

pub trait Deserialize: Sized {
    fn deserialize<D: Decode + ?Sized>(decoder: &mut D) -> Result<Self>;

    fn deserialize_many<D: Decode + ?Sized>(decoder: &mut D, count: usize) -> Result<Vec<Self>> {
        let mut out = Vec::with_capacity(count.min(4096));
        for _ in 0..count { out.push(Self::deserialize(decoder)?); }
        Ok(out)
    }
}
```

The generic `[T]::serialize` and `Vec<T>::deserialize` impls dispatch through these methods:

```rust,ignore
impl<T: Serialize> Serialize for [T] {
    fn serialize<E: Encode + ?Sized>(&self, encoder: &mut E) -> Result<()> {
        encoder.write_varint_u64(self.len() as u64)?;
        T::serialize_slice(self, encoder)   // ← dispatches through T's batch path
    }
}

impl<T: Deserialize> Deserialize for Vec<T> {
    fn deserialize<D: Decode + ?Sized>(decoder: &mut D) -> Result<Self> {
        let declared = decoder.read_varint_u64()?;
        let len = guard_element_count::<T, _>(declared, decoder)?;
        T::deserialize_many(decoder, len)   // ← dispatches through T's batch path
    }
}
```

And `u8` overrides the defaults with single bulk operations:

```rust,ignore
impl Serialize for u8 {
    fn serialize<E: Encode + ?Sized>(&self, encoder: &mut E) -> Result<()> {
        encoder.write_byte(*self)
    }
    fn serialize_slice<E: Encode + ?Sized>(slice: &[u8], encoder: &mut E) -> Result<()> {
        encoder.write_bytes(slice)   // ← one extend_from_slice / write_all
    }
}

impl Deserialize for u8 {
    fn deserialize<D: Decode + ?Sized>(decoder: &mut D) -> Result<Self> {
        decoder.read_byte()
    }
    fn deserialize_many<D: Decode + ?Sized>(decoder: &mut D, count: usize) -> Result<Vec<u8>> {
        let mut out = vec![0u8; count];
        decoder.read_into(&mut out)?;   // ← one bounds-checked memcpy / read_exact
        Ok(out)
    }
}
```

That's the whole optimisation. No `unsafe`, no specialisation feature, no wire-format change. Every other `Vec<T>` decode still takes the existing per-element loop unchanged — the default impl preserves v0.5 behaviour. Only `u8` (and any user type that opts into overriding) takes the bulk path.

The same trick applies to encode: `Vec<u8>` and `&[u8]` now serialise via one `extend_from_slice` instead of N `push` calls.

## Plus a varint hot-path tweak

Single-byte fast path for the most common varint case (values < 128 — every length prefix that fits in 7 bits, every small integer):

```rust,ignore
// In Encode::write_varint_u64
if value < 0x80 {
    return self.write_byte(value as u8);   // skip the stack-buffer round-trip
}
// ... multi-byte path

// In Decode::read_varint_u64
let first = self.read_byte()?;
if first < 0x80 {
    return Ok(u64::from(first));   // skip the loop overhead
}
// ... multi-byte path
```

Smaller win on its own (~13 % on `u64` round-trip) but broadly applicable.

## Full comparative results

All numbers Criterion medians, Windows x86_64, Rust stable release build, project release profile. Reproduce with `cargo bench --bench comparative --features derive`.

### Wins

| Workload | pack-io | nearest competitor | pack-io advantage |
|---|---:|---|---|
| `Vec<u8>` 4 KiB decode | **59 ns** | bincode 76 ns | **+22 %** |
| Owned struct decode | **161 ns** | rkyv 154 ns | ~tied (+5 % over bincode) |
| Zero-copy view of 64-byte `&str` | **5.0 ns** | uncontested | bincode / postcard have no zero-copy story |

### Losses (and why)

| Workload | pack-io | winner | gap | reason |
|---|---:|---|---|---|
| Owned struct encode | 136 ns | bincode 39 ns | 3.5× | bincode's derive emits tighter inline encode codegen with no per-field method dispatch — post-1.0 target |
| `u64` round-trip | 27 ns | bincode 22 ns | 1.2× | Same root cause |
| 64-byte `String` owning | 76 ns | bincode 50 ns | 1.5× | bincode skips the `Config::max_alloc` defence pack-io enforces against hostile length prefixes — worth the 25 ns |
| View vs rkyv archived | 37 ns | rkyv 12 ns | 3× | rkyv's archive is a raw memory layout; pack-io walks varints by spec. We will never match rkyv on this axis unless we ship an archive format too — and the entire wire spec is built on varints precisely to make a third-party implementer possible from one page |

Full per-row analysis in [`docs/PERFORMANCE.md`](https://github.com/jamesgober/pack-io/blob/main/docs/PERFORMANCE.md).

## Breaking changes

**None.** Every v0.5 source file compiles unchanged, every v0.5 wire payload decodes identically. The new trait methods have default implementations; existing `Serialize` / `Deserialize` impls continue to work without modification. The optimisations are opt-in via override — types that don't override the new methods take the v0.5 code path.

## What's next

- **v0.7.0 — Hardening + API freeze.** `cargo-fuzz` continuous harness against the decoder, hostile-input sweep (huge nested types, recursion-bomb structures, length prefixes near `usize::MAX`), cross-platform byte-equivalence verification. Public API freeze (already true since v0.5 — formalised in v0.7). The post-1.0 encode-codegen optimisation against bincode's 3.5× gap is tracked but does not block hardening.
