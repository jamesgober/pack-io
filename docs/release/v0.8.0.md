# pack-io v0.8.0 — Alpha: integration window open

**The first-consumer release.** v0.8.0 carries zero new public API and zero wire-format changes. What it adds is the documentation, status markers, and consumer-shape test / example coverage that signal "pack-io is ready for the first real downstream crates to wire it up". Point releases in the v0.8.x line will track bugs surfaced by real consumer integration; the v0.7 API freeze holds throughout.

## Status

| Field           | Value                                                                                  |
|-----------------|----------------------------------------------------------------------------------------|
| Crate status    | **alpha — integration window open**                                                    |
| Public API      | **frozen since v0.7.0** (no source-breaking change until v2.0)                          |
| Wire format     | **frozen at v0.3.0** (spec version 1.2; no wire change in v0.8.x)                       |
| Beta target     | v0.9.0, gated on a stable stretch with no outstanding bugs                              |
| RC target       | v0.9.5+, critical fixes + doc polish only                                               |
| Stable target   | v1.0.0                                                                                  |

The first real consumers — [`network-protocol`](https://github.com/jamesgober/network-protocol), [`wire-codec`](https://github.com/jamesgober/wire-codec), Hive DB, `raft-io` (log entries), `event-stream` (framing) — start wiring pack-io in during the v0.8.x window. Bugs they surface become v0.8.1 / v0.8.2 / ... point releases. The frozen public API holds; any v0.8.x release that adds API surface is a defect, not a feature.

## What landed

### Eight consumer-shape integration tests

[`tests/integration_scenarios.rs`](https://github.com/jamesgober/pack-io/blob/main/tests/integration_scenarios.rs) exercises the substrate the way real consumers will — not unit-testing primitives in isolation, but walking through end-to-end scenarios that mirror production code:

| Scenario | What it demonstrates |
|---|---|
| `length_framed_message_exchange_round_trips_three_messages` | Sender writes varint-length-prefixed frames into a buffer; receiver pulls each frame off in sequence. The pattern every network protocol uses to delimit messages. |
| `versioned_handshake_round_trips_within_a_version` | Same-revision handshake (control case). |
| `versioned_handshake_v1_client_decoded_by_v2_server_defaults_region` | Forward compat: old client → new server, new field defaults to `None`. |
| `versioned_handshake_v2_client_decoded_by_v1_server_drops_region` | Backward compat: new client → old server, length-framed body skips the v2-only field cleanly. |
| `streaming_event_log_writes_and_reads_a_full_session` | `IoEncoder<BufWriter>` + `IoDecoder<BufReader>` round-trip of seven `Event` enum values across four variant shapes. The WAL / message-queue / replay-buffer pattern. |
| `zero_copy_request_inspection_avoids_per_field_allocation` | `decode_view` round-trip of a request struct, with a **runtime pointer-equality assertion** confirming the view's `&str` / `&[u8]` fields point inside the original wire buffer — not into freshly-allocated heap memory. |
| `tight_max_alloc_protects_decoder_from_oversized_payloads` | `Config::with_max_alloc(16 * 1024)` rejects a 100 MiB length prefix before allocating a single byte. The pattern every public-facing service should use. |
| `encode_into_decode_from_round_trip_through_cursor` | Single-shot helpers over arbitrary `Read` / `Write` (`Cursor<Vec<u8>>` stands in for any transport). |

These don't replace the proptest sweep — they sit alongside it as the "is the substrate ergonomic for real use" canary.

### Two consumer-shape examples

[`examples/protocol_handshake.rs`](https://github.com/jamesgober/pack-io/blob/main/examples/protocol_handshake.rs) runs all four `v1 ↔ v2` cross-decode combinations of a real handshake message and prints what each side observes. Output excerpt:

```
peek_version (server-side dispatch)
  v1 client wire reports version 1
  v2 client wire reports version 2

v1 client → v2 server (forward compat)
  decoded as v2: HandshakeV2 { client_id: 1, capabilities: ["tls"], region: None }
  region field defaulted to None (v1 never wrote it)

v2 client → v1 server (backward compat)
  decoded as v1: HandshakeV1 { client_id: 2, capabilities: ["tls", "h2"] }
  v2-only `region` field is invisible to v1 server (skipped via body length)
```

[`examples/event_log.rs`](https://github.com/jamesgober/pack-io/blob/main/examples/event_log.rs) writes a session of seven events to a tempfile via `IoEncoder<BufWriter<File>>`, then replays them via `IoDecoder<BufReader<File>>` with per-variant dispatch:

```
writing 7 events to /tmp/pack-io-event-log.bin
wrote 103 bytes

replaying the log:
  [1000000] CONNECT     10.0.0.5
  [1000001] HEARTBEAT
  [..]  MESSAGE     10.0.0.5 → 10.0.0.10  (5 bytes)
  ...
  [1000004] DISCONNECT  10.0.0.5   (idle timeout)
```

## Breaking changes

**None.** Every v0.7 source file compiles unchanged. Every v0.7 wire payload decodes identically.

## What v0.8.x will track

The v0.8.x point releases serve a specific purpose: **bugs surfaced by real-consumer integration**. Anything outside that scope waits for v0.9 / v1.0 / v2.0:

| Allowed in v0.8.x                         | Deferred                                |
|-------------------------------------------|-----------------------------------------|
| Bug fixes from real consumers             | New public API (waits for v2.0)         |
| Tightening existing tests                 | New wire format anything                |
| More fuzz targets / hostile inputs        | Source-breaking refactors               |
| Doc improvements                          | Speculative additive features (YAGNI)   |
| Performance fixes (against existing surface) | New feature flags                    |

## What's next

- **v0.8.x point releases** track bugs surfaced by real-consumer integration as they appear. No scheduled cadence — driven by consumer reports.
- **v0.9.0 — Beta.** Promotes pack-io from alpha once a stable stretch passes with no outstanding bugs from the v0.8.x line. Bug-fixes only from this point onward.
- **v0.9.5+ — RC.** Critical fixes + doc polish only.
- **v1.0.0 — Stable.** API + wire format frozen for the entire 1.x line.
